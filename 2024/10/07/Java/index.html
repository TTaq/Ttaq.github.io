Java并发编程

基础

并行与并发

并行指的是多个CPU同时执行多个任务,每个核心实际上可以在同一时间独立执行不同的任务
并发指的是系统由处理多个任务的能力,但是在任意时刻只有一个任务在执行。在单核处理器上,多个任务是通过时间片轮转的方式实现的。

线程安全

指一段代码块或者一个方法在多线程环境下能够被多个线程正确的处理共享数据,那么就是线程安全的
三个要素保证线程安全：
1.原子性:确保当某个线程修改共享变量时,没有其他线程可以同时修改这个变量,即这个操作是不可分割的,可以通过互斥锁或者原子操作来保证
2.可见性:确保一个线程对共享变量的修改可以立即被其他线程看到,volatile关键字可以保证共享变量的修改对所有线程立即可见,并且防止编译器优化导致的可见性问题
3.活跃性问题:确保线程不会因为死锁,饥饿锁,活锁等问题导致无法继续执行

进程,线程和协程

进程指的是应用程序的一个实例,是操作系统分配资源的最小单位,每个进程都有自己的地址空间,全局变量,堆栈和文件描述符等资源
线程是CPU分配资源的基本单位,一个进程中可以包含多个线程,多个线程共用进程的内存空间,共享变量和文件描述符,但是每个线程都持有自己的程序计数器和栈
协程通常被视为比线程更加轻量级的并发单元,主要在一些支持异步编程模型的语言中得到了原生支持

线程的共享内存

线程之间通信一般可以通过消息通信或者共享内存两种方式,Java中采取的是共享内存的方式
线程之间的共享变量存储在主内存中,每个线程都有一个私有的本地内存,本地内存中存储了共享变量的副本。
线程A和线程B之间的通信过程:线程A把本地内存A的共享变量刷新到主内存中,线程B从主内存中读取到线程A刷新过的共享变量,再同步到自己的共享变量副本中

线程的几种创建方式

1.继承Thread类,重写run()方法,调用start()方法启动线程
缺点:Java不支持多重继承,如果类已经继承了另一个类,就不能使用这种方法
2.实现Runable接口,重写run()方法,然后创建Thread对象,将Runable对象作为参数传递给Thread对象,调用start()方法启动线程
优点:可以避免Java的单继承机制,并且更加符合面向对象的编程思想,因为Runable接口将任务代码和线程控制的代码解耦了
3.实现Callable接口,重写call()方法,然后创建FutureTask对象,参数为Callable对象。然后创建Thread对象,参数为FutureTask对象,调用start()方法启动线程

一个8GB内存的系统可以创建多少个线程

除了系统本身的内存大小,还需要考虑Java虚拟机的大小
线程在创建的时候会被分配一个虚拟机栈,默认大小为1M
1个8GB内存的系统可以创建810241024KB/1*1024KB = 8192,由于操作系统本身的运行也需要消耗一定的内存,所以实际上可以创建的线程会比8192少

启动一个Java程序,里面有哪些线程

main线程:Java程序启动时由JVM创建,程序的入口
Reference Handler:用于处理引用对象,负责清理被JVM回收的对象
Finalizer:终结器线程,负责调用对象的finalize方法,对象在垃圾回收器标记为可回收之前,由该线程执行finalize方法,用于执行指定的资源释放操作
Signal Dispatcher:信号调度线程,处理来自操作系统的信号,将他们转发给JVM进一步处理,例如响应中断,停止等信号
Monitor Ctrl-Break:监视器线程,通常由一些特定的IDE创建,用于在开发过程中监控和管理程序执行或者处理中断

调用start()方法会执行run()方法,为什么不直接调用run方法

当调用start()方法时,会创建一个新线程并在新线程中执行run()方法
如果直接调用run方法,则会在当前线程下执行run()方法,没有新的线程被创建,也就没有实现多线程的效果

线程常用调度方法

等待

1.wait():当一个线程A调用一个共享变量的wait()方法时,线程A会被阻塞挂起,
直到线程B调用了共享变量的notify或者notifyAll()方法或者其他线程调用了线程A的interrupt(),线程A抛出InterruptedException异常返回
2.wait(long timeout):该方法比wait()多了一个超时参数,如果调用共享变量的wait()方法后，没有在指定时间被唤醒，那么就会因为超时而返回
3.wait(long timeout,int nacos),其内部调用的是wait(long timeout)

通知

1.notify():一个线程A调用共享变量的notify()后,会唤醒一个在这个共享变量上调用wait()系列方法后被挂起的线程。
一个共享变量上可能有多个线程挂起，具体唤醒的线程是随机的
2.notifyAll():会唤醒所有在该共享变量上调用wait系列方法后被挂起的线程
3.Thread中还有一个join()方法,如果一个线程A执行了thread.join(),当前线程A会等待thread线程终止之后才从thread.join()返回

让出优先权

Thread类的静态方法,当一个线程调用yield()方法，是在提示线程调度器,当前线程请求让出自己的CPU,但是线程调度器可能会忽略这个提示

休眠

sleep(long mills):Thread类中的静态方法,当一个执行中的线程调用sleep方法后,线程A会暂时让出指定时间的执行权
但是线程A中的监视器资源,如锁，还是持有不让出的,指定睡眠时间到了之后线程后接着参与CPU的调度,获取到CPU资源后就可以继续运行

中断

Java中线程终端是一种线程间的协作模式,通过设置线程的中断标志并不能直接终止该线程的执行,被中断的线程会根据中断状态自行处理
void interrupt():中断线程,当线程A运行时,线程B可以调用interrupt()来设置线程的中断标志为true并立即返回.设置标志仅仅是设置标志,线程B实际没有中断,会继续执行下去
boolean isInterrupted():检测当前线程是否被中断
boolean interrupted():检测当前线程是否被中断,如果发现该线程被中断,则会清除中断标志

线程有几种状态

NEW:当线程被创建后,比如使用new Thread(),他处于新建状态,此时,线程已经被分配了必要的资源,但是还没有开始执行
RUNABLE:当调用线程的start()方法时,线程进入可运行状态,在这个状态下,线程可能正在运行也可能在等待CPU的时间片,具体取决于线程调度器的调度策略
BLOCKED:线程试图获取一个锁进入同步块/方法时,如果该锁被其他线程持有,那么该线程就会进入阻塞状态,直到获取锁为止
WAITING:线程进入等待状态是因为调用了Object.wait()方法或LockSupport.park()方法。在等待状态下,线程需要被其他线程显示唤醒,否则不会自动执行
TIME_WAITING:当线程调用了带有超时参数的方法时,如:Thread.sleep(long mills),Object.wait(long timeout),该线程进入超时等待状态,线程在指定时间结束后会自动返回可运行状态
TERMINATED:当线程的run()方法执行完,或者在run方法的过程中因未捕获的异常终止了执行,线程进入终止状态。一旦线程终止,他的生命周期结束,不能再被重新启动

什么是线程上下文切换

使用多线程的目的是为了充分使用CPU,但是我们知道,并发其实是一个CPU去处理多个线程
上下文切换:
CPU的资源分配采用了时间片轮转的方式,也就是给每个线程分配一个时间片,线程在时间片内占用CPU执行任务,当线程使用完CPU后,就会处于就绪状态并让出CPU让其他线程占用

线程可以被多核调度吗

可以
操作系统的调度器负责把线程分配给可用的CPU核心,从而实现并行处理
多核处理器提供了并行执行多个线程的能力,每个核心可以独立执行一个或多个线程.操作系统的任务调度器会根据策略和算法决定哪个线程何时在哪个处理器上运行

守护线程

Java的线程主要分为守护线程和用户线程
当从main方法启动一个应用程序时,main方法所在的线程就是一个用户线程,而在JVM内部同时也会启动很多守护线程,比如垃圾回收线程
当最后一个非守护线程结束后,JVM就会正常退出,而守护线程是否结束不影响JVM的退出

线程间的通信方式

共享对象

多个线程可以访问和修改同一个对象实现信息的传递,比如volatile和synchronized关键字
volatile:用于修饰成员变量,告知程序任何对该变量的访问均需要从共享内存中进行获取,所有对该变量的修改也需要重新刷新到共享内存中,保证所有线程对该变量访问的可见性
synchronized:可以修饰方法,或者以同步代码块的方式来使用.确保多个线程在同一时刻只有一个线程在执行某个方法或某个代码块

wait()和notify()

一个线程调用wait()时,他会进入该对象的等待池,并释放已经持有的该对象的锁,进入等待状态,直到其他线程调用相同对象的notify()或notifyAll()
一个线程调用notify()时,他会唤醒在对象等待池中的任一线程,使其进入锁池,等待获取锁
通常与锁(特别时ReentrantLock)配合使用,为线程提供了一种等待某个值为真的机制,并允许其他线程在该条件变化时通知等待线程

Exchanger

Exchanger是一个同步点,可以在两个线程之间交换数据.一个线程调用exchange(),将数据传递给另一个线程,同时接收另一个线程的数据

CompletableFuture

是Java8引入的一个类,支持异步编程,允许线程在完成计算后将结果传递给其他线程

sleep和wait的区别

sleep()和wait()是Java中暂停线程的两个方法,sleep是让当前线程休眠,不涉及对象类,也不需要获取对象的锁,属于Thread类方法.
wait是让获得对象锁的线程进行等待,前提是该线程获得对象的锁,属于Object类的方法

1.所属类不同,sleep属于Thread类,wait属于Object类
2.锁行为不同,当线程执行sleep方法时,它不会释放任何锁,当线程执行wait方法时,它会释放它持有的那个对象锁,使得其他线程有机会可以获取到该对象的锁
3.使用条件不同:sleep可以在任何地方调用,wait必须在同步代码块或同步方法内调用,因为调用wait的前提是当前线程必须持有对象锁
4.唤醒方式不同:
调用sleep方法后线程会进入TIME_WAIT状态,在指定时间内暂停执行,当指定时间结束后,线程会自动转化到RUNABLE状态,等待CPU再次调度
调用wait方法后,线程会进入WAIT状态,直到有其他线程在同一对象上调用notify()或notifyAll(),线程才会从WAIT转化为RUNABLE状态,准备再次获得CPU的执行权

线程安全使用场景

线程安全是指在多个线程的环境下,多个线程对共享数据的访问不会导致数据的不一致性
常见场景:单例模式
单例模式确保一个类只有一个实例,并提供一个全局访问点,在多线程环境下,如果多个线程同时尝试创建实例,单例类确保只创建一个实例
饿汉式:在类加载就初始化单例来保证线程安全

懒汉式:在第一次使用时初始化,使用双重检查锁来保证线程安全,volatite保证可见性,synchronized保证同步性

ThreadLocal

ThreadLocal是什么

ThreadLocal是Java中用来实现线程局部变量的工具类.它允许每个线程都拥有自己的独立副本,从而实现线程隔离,用于解决多线程中共享对象的问题
在Web应用中,可以使用ThreadLocal来存储用户会话信息,这样每个线程在处理用户请求时都可以方便地获取到当前用户的会话信息
在数据库操作中,可以使用ThreadLocal存储自己的数据库连接对象,每个线程都有自己独立的数据库连接,从而避免了多线程竞争同一数据库连接的问题
在格式化操作中，可以使用ThreadLocal存储SimpleDateFormat实例,避免多线程共享同一实例导致的线程安全问题

除了ThreadLocal外,解决线程安全问题的方法

1.synchronized关键字可用于代码块,确保同一时间只有一个线程可以执行特定的代码块
2.Java并发包中提供了Lock接口和一些实现类,如:ReentrantLock,提供了公平锁和非公平锁
3.Java并发包还提供了一组原子变量类,利用CAS机制实现无锁的原子操作,适用于简单的计数器场景
4.volatile修饰变量保证变量的可见性,修改操作是立即同步到主存的,读操作是在主存中读取

工作中ThreadLocal的使用场景

1.用于存储用户信息,假如在服务层和持久层都需要用到用户信息,可以在控制层对用户请求进行拦截,并且把用户信息存储到ThreadLocal中
2.用于记录操作日志,存储变更前后对象

ThreadLocal的实现

ThreadLocal内存泄露

随着线程Thread的结束,其内部的ThreadLocalMap也会被回收,从而避免内存泄露
但是如果一个线程一直执行,且ThreadLocalMap中的Entry.value一直指向某个强引用对象,那么这个对象也就不会被回收,从而导致内存泄露
如何解决: remove会将ThreadLocalMap中的所有key为null的Entry都清除掉,这样就避免了内存泄露问题
key为什么要设置为弱引用:当内存不足的时候,JVM会主动回收掉弱引用对象

Java内存模型

锁

乐观锁与悲观锁

悲观锁:在修改数据之前将其锁定,修改完成后将锁释放,以确保数据的一致性和完整性
常见的悲观锁:synchronized,ReentrantLock类
适用场景:
高并发且数据竞争激烈的场景
数据一致性要求极高的场景
写操作频繁的场景
事务执行时间较长的场景
优点：数据一致性高,简单易容
缺点:性能开销大,容易造成死锁,可能导致资源浪费

乐观锁:基于版本号和时间戳来实现,在每次更新数据的时候,先获取当前数据的版本号或时间戳,然后在更新时对比版本号或时间戳是否一致,若一致则更新成功,如果不一致,证明该数据被其他线程修改过,更新失败
常见的乐观锁:Atomic类
核心算法:CAS,通过比较内存中的值是否和预期中的值相等来判断是否冲突,如果存在,则返回失败；如果不存在,则执行更新操作
适用场景:写操作比较少,数据冲突比较少,重试成本较低,系统能够容忍一定程度的失败
优点：高并发高吞吐,无锁操作,无死锁风险
缺点:冲突处理复杂,数据一致性风险,需要额外字段,处理不当造成死循环风险

注意事项:采用乐观锁更新数据库数据时,需要注意事务隔离级别问题,应该使用读已提交,
如果使用的是可重复读,且通过额外字段version来实现乐观锁更新,那么在乐观锁冲突更新失败后自旋时因为MVCC机制查询到的数据是一个副本值,就算别的事务更新成功了读取到的version都是历史值,这样会导致死循环递归最后栈溢出
如果采用作修改的字段进行判断更新,如果在锁冲突频繁的时候使用可重复读事务隔离级别可能会出现持有锁时间过长问题,因为在可重复读隔离级别下,在下一个事务中执行更新语句,就算where条件不成立也会将这一行的数据进行锁定,需要等待事务提交或回滚才会释放锁,也就是说在自旋的时候其他事务时,想要更新数据等待时间会变长,影响系统吞吐量,而使用读已提交级别时,where中条件不成立更新失败时不会持有锁,也就是说在事务A更新失败自旋时,事务B可以对该数据进行更新,而不需要等待事务A自旋更新成功
